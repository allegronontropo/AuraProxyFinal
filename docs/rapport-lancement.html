2. Presentation du projet

2.1 Qu'est-ce qu'Aura Proxy ?

Aura Proxy est une plateforme middleware qui se positionne entre les applications clientes et les fournisseurs d'intelligence artificielle (OpenAI, Anthropic, Google, Mistral). Elle offre :

  • Une API unifiee compatible OpenAI pour acceder a tous les fournisseurs
  • Un cache semantique pour eviter les appels redondants et reduire les couts
  • Un controle budgetaire par projet avec alertes automatiques
  • Un tableau de bord pour le suivi en temps reel de la consommation

2.2 Inspiration : Portkey.ai

Le projet s'inspire de Portkey.ai, une plateforme SaaS enterprise qui propose un AI Gateway unifie, de l'observabilite, des guardrails et de la gestion de prompts pour plus de 1600 modeles LLM.

Aura Proxy reprend les concepts fondamentaux de Portkey (gateway unifie, cache, cout tracking, RBAC) tout en se differenciant par :

Portkey.ai Aura Proxy
SaaS payant (a partir de $49/mois) Self-hosted, open-source
1600+ fournisseurs LLM 4 fournisseurs (MVP scope stage)
Architecture Go/Node proprietaire Architecture NestJS modulaire (pedagogique)
Dashboard SaaS Dashboard Next.js integre

5.1 Pourquoi NestJS et pas Express/Fastify brut ?

Le choix initial etait Fastify brut. Apres analyse, NestJS a ete retenu car il fournit nativement les abstractions necessaires :

  • Modules : organisation du code par domaine metier (Auth, Chat, Cache...)
  • Injection de dependances : les services sont injectes, jamais instancies manuellement
  • Guards : verifient l'autorisation AVANT que la requete atteigne le controleur
  • Interceptors : ajoutent du comportement (logging, retry) AUTOUR du handler
  • Swagger : documentation API generee automatiquement depuis les decorateurs

NestJS utilise Fastify en interne comme adaptateur HTTP, conservant ainsi ses performances.

6. Architecture retenue

6.1 Architecture globale

Le projet adopte une architecture monorepo avec separation stricte entre le proxy backend (NestJS), le dashboard frontend (Next.js) et les packages partages :

aura-proxy/
  apps/
    proxy/         NestJS 11 — API Gateway LLM
    dashboard/     Next.js 15 — Dashboard admin/client
  packages/
    db/            Prisma client (Singleton)
    redis/         Redis client (Singleton)
    shared/        Types, constantes, utilitaires

6.2 Modules NestJS

Le proxy backend est organise en 11 modules NestJS, chacun responsable d'un domaine :

Module Responsabilite Design Pattern
ConfigModule Variables d'environnement —
PrismaModule Connexion base de donnees Singleton
RedisModule Connexion Redis Singleton
HealthModule Endpoint de sante —
AuthModule Validation des cles API —
BudgetModule Controle budgetaire Observer
ProvidersModule Fournisseurs LLM Strategy
ChatModule Endpoint principal —
CacheModule Cache semantique —
StreamingModule Reponses SSE —
EventsModule Bus d'evenements Observer

6.3 Flux d'une requete

Chaque requete entrante traverse une chaine de responsabilites :

  1. AuthGuard — Verifie la cle API (SHA-256 + cache Redis)
  2. BudgetGuard — Verifie que le budget du projet n'est pas depasse
  3. RateLimiterGuard — Verifie la limite de requetes (sliding window)
  4. CacheInterceptor — Cherche une reponse similaire dans le cache semantique
  5. ChatService — Si cache MISS, appelle le fournisseur LLM via le Strategy Pattern
  6. CostTrackerInterceptor — Enregistre le cout et les metriques
  7. EventEmitter — Emet les evenements pour les workers asynchrones

8.3 Lancement du projet en local

# 1. Cloner le dépôt
git clone https://github.com/allegronontropo/Aura-Proxy.git
cd Aura-Proxy

# 2. Démarrer les services Docker
docker compose up -d

# 3. Installer les dépendances
npm install

# 4. Générer le client Prisma
npm run db:generate

# 5. Appliquer les migrations
npm run db:migrate -- --name init

# 6. Lancer en développement
npm run dev

9.1 Résultats attendus

À l'issue des 13 semaines, les livrables attendus sont :

  • Une API Gateway fonctionnelle capable de proxier les requêtes vers 4 fournisseurs LLM
  • Un cache sémantique réduisant les coûts d'au moins 30% sur les requêtes récurrentes
  • Un tableau de bord complet avec authentification, analytics et gestion des clés API
  • Une intégration Stripe opérationnelle avec 3 plans tarifaires
  • Une documentation technique (Swagger + README) et une couverture de tests unitaires ≥ 70%
  • Un déploiement Docker reproductible en une commande

La plus-value par rapport à l'existant réside dans le fait qu'aucune solution open-source en TypeScript ne propose actuellement l'ensemble cache sémantique + contrôle budgétaire + dashboard unifié. Les solutions existantes (Portkey, Helicone) sont soit payantes, soit limitées en fonctionnalités.
